title: Regeneration

# Regeneration
```

Opening paragraph: After your first `handover generate`, you do not need to regenerate everything from scratch. The `--since` flag lets you re-analyze only files that changed since a git ref.

**Sections (H2 level):**

1. **Incremental regeneration** — cover the `--since` flag:
   - Usage: `handover generate --since HEAD~3` or `--since main` or `--since v1.0`
   - What it does: detects files changed since the ref using `git diff` + `git status`, only re-analyzes those files, uses cached round results for unchanged files
   - When to use: after making targeted changes, to save time and API cost
   - Example commands

2. **Cache behavior** — explain how the round cache works:
   - Location: `.handover-cache/` directory in the project root
   - Content-hash mode (default): each round's input is hashed; if the hash matches a cached result, the round is skipped
   - `--no-cache` flag: discard cached results and run all rounds fresh
   - How `--since` interacts with the cache: changed files produce new context hashes, triggering fresh AI rounds; unchanged files reuse cached results

3. **Non-git fallback** — explain graceful degradation:
   - Not a git repo: prints a warning, falls back to full content-hash mode (all files analyzed)
   - Detached HEAD: prints a warning, falls back (cannot compute diff without a branch)
   - Shallow clone: prints a warning, falls back (insufficient history for diff)
   - Invalid ref: throws a clear error with the bad ref name
   - Table summarizing fallback behavior by environment

4. **Flags reference** — table:
   | Flag | Description |
   |------|-------------|
   | `--since <ref>` | Only re-analyze files changed since this git ref |
   | `--no-cache` | Discard cached results and re-run all rounds |

5. **Next steps** — cross-links to:
   - [getting-started](/user/getting-started/) for first-run workflow
   - [search](/user/search/) for querying generated docs
   - [configuration](/user/configuration/) for cache and output settings

# Testing
```

Opening paragraph: handover uses Vitest for unit tests with a set of established mock patterns for isolating provider and filesystem dependencies. This guide covers the key patterns and conventions.

**Sections (H2 level):**

1. **Running tests** — brief recap (reference `development.md` for full setup):
   - `npm test` runs all unit tests
   - `npm test -- --coverage` for coverage reports
   - `npm test -- src/path/to/file.test.ts` for individual files

2. **Mock provider pattern** — document `createMockProvider()`:
   - Location: `src/providers/__mocks__/index.ts`
   - What it does: returns a fully type-checked `LLMProvider` stub with `vi.fn()` on each method (`complete`, `estimateTokens`, `maxContextTokens`)
   - Why it exists: real providers require API keys, network, and SDK constructors; the mock enables round-level unit testing without any provider setup
   - Usage example (inline code block):
     ```typescript
     import { createMockProvider } from '../providers/__mocks__/index.js';

     const provider = createMockProvider();
     (provider.complete as ReturnType<typeof vi.fn>).mockResolvedValueOnce({
       data: { /* round schema output */ },
       usage: { inputTokens: 100, outputTokens: 50, cacheReadTokens: 0, cacheCreationTokens: 0 },
       model: 'mock',
       duration: 0,
     });
     ```
   - Override pattern: `createMockProvider({ name: 'custom-mock' })` to customize individual methods

3. **In-memory filesystem (memfs)** — document the `memfs` + `vol` pattern:
   - Package: `memfs` (already in devDependencies)
   - Purpose: replace `node:fs` and `node:fs/promises` with an in-memory filesystem for test isolation
   - Setup pattern (inline code block showing `vi.mock` for both `node:fs` and `node:fs/promises`, `vol.reset()` in `beforeEach`, and `vol.fromJSON` for seeding):
     ```typescript
     import { vol } from 'memfs';

     vi.mock('node:fs/promises', async () => {
       const memfs = await import('memfs');
       return memfs.fs.promises;
     });

     vi.mock('node:fs', async () => {
       const memfs = await import('memfs');
       return memfs.fs;
     });

     beforeEach(() => {
       vol.reset();
     });

     // Seed the virtual filesystem
     vol.fromJSON({
       '/mock-home/.handover/credentials.json': JSON.stringify({ token: 'abc' }),
     });
     ```
   - Reference file: `src/auth/token-store.test.ts` is the canonical example

4. **Coverage exclusions** — explain the frozen exclusion list:
   - Location: `vitest.config.ts`, `coverage.exclude` array
   - Policy: the list is frozen with a `Last frozen:` date. Every entry has a written justification comment. Do NOT add entries without justification. Do NOT enable `thresholds.autoUpdate`.
   - Categories (brief summary, not exhaustive list):
     - CLI entry points (integration-only, require full pipeline)
     - Provider SDK wrappers (require real SDKs / network / API keys)
     - Semantic search runtime (require SQLite + embeddings infrastructure)
     - MCP server runtime (require full MCP SDK server lifecycle)
     - Static analyzers (require real filesystem and git context)
   - Current thresholds: 90% lines/functions/statements, 85% branches
   - Why some files ARE included despite being complex: `mcp/tools.ts`, `mcp/errors.ts`, and `mcp/http-security.ts` are intentionally testable and not excluded

5. **Next steps** — cross-link to:
   - [development](/contributor/development/) for full setup and PR workflow
   - [architecture](/contributor/architecture/) for codebase structure
