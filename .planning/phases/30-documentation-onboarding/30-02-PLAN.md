**Page structure:**

```markdown
---
title: Search

# Search
```

Opening paragraph: handover generates a semantic search index alongside documentation. Use `handover search` to query it and `handover reindex` to build or update it.

**Sections (H2 level):**

1. **Building the index** — explain `handover reindex` as the first step after `handover generate`. Cover:
   - Basic invocation: `handover reindex`
   - What it does: scans generated docs, chunks them, generates embeddings, stores in a local SQLite vector index
   - `--force` flag: re-embed all documents, ignoring change detection
   - `--embedding-mode` flag: `local-only`, `local-preferred`, `remote-only`
   - Change detection: reindex only re-embeds documents whose content hash changed since the last run
   - Example commands with code blocks

2. **Searching** — explain `handover search` with fast and QA modes:
   - Basic: `handover search "authentication"`
   - Fast mode (default): retrieval-only semantic similarity, returns ranked results with relevance scores
   - QA mode: `handover search "How does auth work?" --mode qa` — retrieves relevant chunks then synthesizes an answer via the configured LLM provider (requires an API key)
   - Output format: rank, relevance percentage, source file, section path, snippet

3. **Filtering results** — cover CLI flags:
   - `--type <type>`: filter by document type (repeatable). Valid types come from the generated docs (e.g., `architecture`, `modules`, `features`, etc.)
   - `--top-k <n>`: number of results (default: 10)
   - `--embedding-mode <mode>`: override embedding locality for this search
   - Table format for all flags

4. **Quality signals** — explain what the output tells you:
   - Zero results + empty index: suggests running generate then reindex
   - Zero results + populated index: suggests refining query, broadening type filters, increasing top-k
   - Low relevance warning: distance warning when best match exceeds threshold
   - TTY vs piped output: clickable OSC8 links on TTY, plain paths when piped

5. **Next steps** — cross-links to:
   - [configuration](/user/configuration/) for embedding settings
   - Do NOT include a link to `/user/regeneration/` yet — that page is created in plan 30-03. Plan 30-03 will add the cross-link after creating regeneration.md.

**Tone:** Match `getting-started.md` — direct, practical, no fluff. Use concrete examples. Keep each H2 section short enough to fit on one screen.

**Link format:** Use absolute-from-root links with trailing slashes for internal links (e.g., `/user/configuration/`). This is required by the starlight-links-validator default settings.
